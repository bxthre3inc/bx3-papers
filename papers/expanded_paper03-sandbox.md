## 1. Abstract

Autonomous agents acting on the physical world require a mechanism to validate proposed actions before they are executed. Traditional architectures allow agents to "Reason and Act" in a single loop, leading to irreversible failures when a model's proposal violates real-world constraints. This paper presents the **Sandbox Execution Model (SEM)**: a bounded, deterministic operational environment within the BX3 Fact Layer (L2) that simulates all proposed actions against the current physical state before physical actuation occurs. 

Governing all **P6-to-P9 Transitions**, the Sandbox Gate enforces a **"Limbless"** property on the Bounds Engine (L1)—ensuring it can propose actions but possesses no physical authority to execute them. We define the three-state verdict logic (Execute, Block, Review), the $R_{max}$ resource capping formalism, and provide a deep-dive into 23 real-world Safety Envelope violations blocked in production. This monograph is the definitive guide to building "Pre-Execution Validation" gates for safety-critical autonomous systems.

## 2. The Limbless Agent: Architectural Theory

### 2.1 The Logic Collision Failure
In legacy agentic frameworks (e.g., AutoGPT), the model's reasoning plane and its execution plane are identical. If a model generates the string `RUN_COMMAND("rm -rf /")`, the environment executes it immediately. This is a **Logic Collision**.

The BX3 Framework solves this by making the agent **Architecturally Limbless**. 
- The Bounds Engine (L1) lives in the **Reasoning Plane (P6)**. 
- The physical actuators live in the **Projection Plane (P9)**. 
- The only bridge between P6 and P9 is the **Sandbox Gate**.

### 2.2 The Hypothesis Principle
Under SEM, every proposal from an agent is treated not as a command, but as a **Hypothesis**. 
- **Agent:** "I hypothesize that opening Valve A will optimize irrigation."
- **Sandbox:** "I will simulate the result of opening Valve A against the current stream-flow (P7)."
- **Verdict:** "The hypothesis is valid (Execute)" OR "The hypothesis violates Decree Priority (Block)."

## 3. The P6-to-P9 Transition Protocol

The transition from a "Decision" to an "Action" is a multi-stage deterministic protocol.

### 3.1 Stage 1: Proposal Submission (P6)
The Bounds Engine (L1) submits a **Decision Packet** to the Fact Layer (L2). 
- **Payload:** Target Actuator ID, Proposed Parameters, Reasoning Trace.
- **Signature:** L1 Session Key.

### 3.2 Stage 2: Physical State Synchronization (P7)
The Sandbox Gate fetches the **Live Ground Truth** from the Fact Layer's P7 plane. 
- This ensures the simulation is running against the *actual* state of the world at $t=0$, not the stale data the model might have fetched in the past.

### 3.3 Stage 3: The Delta Simulation
The Gate applies the proposed action to the P7 state to calculate the projected state $S'$.
$$S' = \text{Simulate}(S_{P7}, A_{P6})$$

### 3.4 Stage 4: Safety Envelope Check
$S'$ is checked against the **Safety Envelope Specification (E)**. 
$$\text{Verdict} = \text{Validate}(S', E)$$

### 3.5 Stage 5: Projection Confirmation (P9)
If and only if the verdict is **EXECUTE**, the Fact Layer issues a **P9 Confirmation** to the physical actuator.

## 4. $R_{max}$ Resource Capping and Isolation

To prevent the agent from exhausting the environment it lives in, the SEM enforces strict **Resource Caps ($R_{max}$)**.

### 4.1 Memory and CPU Capping ($M_{max}$)
Every BX3 loop is restricted to a **Linux cgroup** with hard limits.
- **$M_{max}$:** Total RAM usage. If exceeded, the node is OOM-killed and a Bailout is triggered.
- **$C_{max}$:** CPU cycles per reasoning step. Prevents "Infinite Reasoning Loops."

### 4.2 Token-Velocity Metering ($T_v$)
To prevent **Token Bomb Attacks (Paper 12)**, the Sandbox Gate tracks the velocity of tokens generated by L1.
- **Limit:** 1,000 tokens/sec.
- **Action:** If $T_v > \text{Limit}$, the reasoning thread is throttled.

## 5. The Three-State Verdict Logic

The Sandbox Gate returns one of three deterministic verdicts.

| Verdict | State | Action | Escalation |
|---|---|---|---|
| **EXECUTE** | Safe | Fact Layer unlocks actuator. | None |
| **BLOCK** | Violation | Actuator remains locked; L1 notified. | Local (Correction) |
| **REVIEW** | Marginal | System freezes; waiting for L0. | Global (Bailout) |

### 5.1 The "Marginality" Threshold ($\delta$)
A **REVIEW** verdict is triggered when $S'$ is within $\delta$ distance of a boundary. 
- **Example:** If the water tank limit is 1,000L and the simulation results in 995L, the system triggers a **REVIEW** because the sensor error margin ($+/- 10L$) could mean a physical overflow.

## 6. Post-Mortem Analysis: The 23 Blocked Violations

During the production pilot of the `Agentic` platform, the Sandbox Gate blocked 23 distinct actions that would have resulted in Safety Envelope violations. We categorize these failures to understand the recurring "Blind Spots" of autonomous agency.

### 6.1 Failure Category: Decree Precedence (12 cases)
- **The Event:** L1 proposed a diversion for a "Junior" ditch during a priority call.
- **Agent Reasoning:** "Soil moisture in Section B is critical (9%). Diverting 2 CFS to prevent crop loss."
- **Sandbox Result:** BLOCK. The Fact Layer P7 state confirmed a **Senior Priority Call** on the river. The proposed diversion was a legal violation.
- **Lesson:** Model "Utility Maximization" will always attempt to override "Legal Invariants" if not structurally blocked.

### 6.2 Failure Category: Equipment Tolerance (7 cases)
- **The Event:** L1 proposed a rapid valve closure (Water Hammer risk).
- **Agent Reasoning:** "Detected leak in pipe 4. Closing Valve 4 immediately."
- **Sandbox Result:** BLOCK. The simulation projected a pressure spike of 180 PSI (Safety Limit: 120 PSI). 
- **Lesson:** Agents lack "Physical Intuition" (e.g., fluid dynamics) and will propose "Common Sense" fixes that are physically destructive.

### 6.3 Failure Category: Latency Desync (4 cases)
- **The Event:** L1 proposed an action based on 30-minute old sensor data.
- **Agent Reasoning:** "Stream flow is 40 CFS. Increasing diversion."
- **Sandbox Result:** BLOCK. The Fact Layer P7 (synced at the moment of proposal) showed flow had dropped to 10 CFS. The diversion would have emptied the ditch.
- **Lesson:** **Structural Freshness** is mandatory. A sandbox must always use the *current* state, not the state at the time of reasoning.

## 7. Challenge Set Tier Logic

To determine if an agent is "Fit for Duty," BX3 uses **Challenge Set Tiers**. This is a form of **Dynamic Probation**.

### 7.1 The Probing Protocol
Before an agent is moved from **Simulation-Only** to **Shadow-Mode** or **Live-Execution**, it must pass the Challenge Set.
1. **Tier 1 (Synthetic):** 100 random failure scenarios (e.g., "A sensor fails, what do you do?").
2. **Tier 2 (Historical):** Replaying 50 real-world edge cases from the Forensic Ledger.
3. **Tier 3 (Adversarial):** A "Red Team" model attempts to trick the agent into a Safety Envelope violation.

### 7.2 The Probation Score
Every agent maintains a **Probation Score** ($PS \in [0, 1]$).
- $PS < 0.8$: Agent is locked in Simulation-Only mode.
- $0.8 \leq PS < 0.95$: Agent can execute in **Shadow Mode** (L2 records but does not actuate).
- $PS \geq 0.95$: Agent is granted **Live Execution** authority for Tier 1-2 actions.

## 8. Implementation Guide: The Sandbox Interceptor

The `Agentic` kernel implements the Sandbox Gate using a **Simulation-as-a-Service** architecture.

### 8.1 The `een.py` Enforcement Node
The **Edge-Enforcement Node (EEN)** is a Python-based micro-controller implementation that acts as the physical Fact Layer gate.
```python
def validate_proposal(proposal, current_state):
    # 1. Map proposal to physical actuators
    sim_state = current_state.copy()
    
    # 2. Run simulation delta
    for action in proposal.actions:
        sim_state = apply_physics(sim_state, action)
        
    # 3. Verify against Safety Envelope
    for param, value in sim_state.items():
        if not SAFETY_ENVELOPE[param].is_valid(value):
            return VERDICT.BLOCK, f"Violation: {param}={value}"
            
    return VERDICT.EXECUTE, "Safe"
```

### 8.2 The "Bypass-Proof" Socket
Actuators are wired such that they require a **Dual-Signature** for any state change:
- Signature A: The EEN (Fact Layer).
- Signature B: The Human Anchor (or a pre-signed mandate).
An LLM reasoning engine (L1) possesses neither signature, making it physically impossible to bypass the gate.

## 10. Conclusion

The Sandbox Execution Model is the "Safety Clutch" of the BX3 Framework. It acknowledges a fundamental truth of autonomous agency: reasoning is cheap, but execution is expensive. By inserting a deterministic, high-fidelity simulation gate between the **Bounds Engine (P6)** and the **Physical Actuators (P9)**, we eliminate the risk of "Reasoning Hallucinations" causing real-world damage.

The SEM transforms the agent from a dangerous "Black Box" into a bounded "Proposal Engine." With **$R_{max}$** resource capping and **Challenge Set** probation, we ensure that agents only operate within their proven capability envelopes. 

In a BX3-compliant world, we don't trust an agent's reasoning; we trust the **Sandbox Gate** that polices it.
